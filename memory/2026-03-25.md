- 已明确：文件多本身不等于成长，只有“真实、可核对、能提炼、能反哺规则”的记录体系才真正有价值

### 自动化与提醒方向
- 用户说明：由于工作忙，倾向于把时间提醒、价格提醒等重复事务尽量自动化
- 已形成共识：自动化的价值在于减少记忆负担、轮询负担和上下文切换，而不是炫技
- 已讨论价格提醒方向：对于用户，最实用的是“到价提醒 / 突破提醒 / 进入区间提醒”三类
- 已讨论 TradingView 路线：TradingView 本身可以监听价格，但 webhook 手动填写过于麻烦
- 用户已明确偏好：后续若做提醒自动化，优先考虑 **Playwright 浏览器自动化方案**，用于减轻 TradingView alert 创建时的重复输入成本
- 当前最小目标已明确：脚本只做“TradingView alert 弹窗出现后，自动填 webhook 地址”，先不自动提交

### 飞书通知现状判断
- 用户观察到：OpenClaw 发到飞书的消息会让飞书 App 图标出现角标，但不会进入 iPhone 通知中心，也不会形成横幅提醒
- 结论：当前链路下，OpenClaw/插件类消息在飞书中更像“会话未读更新”，不等同于普通聊天消息的系统 push
- 已形成判断：如果未来需要真正可靠的强提醒，不应继续假设“普通飞书会话消息天然等于 push”，而应考虑把强提醒单独设计为专门通道
