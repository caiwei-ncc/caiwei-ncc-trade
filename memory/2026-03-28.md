## OpenClaw gateway 故障排查与修复（阿里云）

- 现象：白天多次出现飞书发消息无响应，但早上 09:30 定时提醒仍能正常发出；说明“主动发送链路”和“实时接收链路”不是一回事。
- 证据：凌晨 02:27-03:03 实时收消息正常；14:21、14:22、14:25、15:46 当时没有正常接收记录；16:07 后重新出现 gateway / websocket 建链。
- 结构结论：cron 主动提醒脚本正常；gateway 实时监听链路存在失联/假死问题。
- 修复收口：
  - 实时监听托管收口到 `systemd --user openclaw-gateway.service`
  - 新增 `scripts/gateway-force-restart.sh`，在脚本/cron 环境下显式补齐 `HOME`、`PATH`、`XDG_RUNTIME_DIR=/run/user/0`、`DBUS_SESSION_BUS_ADDRESS=unix:path=/run/user/0/bus`，然后执行 `openclaw gateway stop` + `sleep 5` + `openclaw gateway start`
  - 健康检查脚本 `scripts/openclaw-gateway-healthcheck.sh` 的恢复动作统一改为调用 `gateway-force-restart.sh`
  - 定时预防性重启 cron 改为白天活跃时段每 2 小时执行一次 `gateway-force-restart.sh`
- 当前 cron：
  - `*/5 * * * * /root/.openclaw/workspace/scripts/openclaw-gateway-healthcheck.sh`
  - `0 8,10,12,14,16,18,20,22 * * * /root/.openclaw/workspace/scripts/gateway-force-restart.sh >> /root/.openclaw/workspace/logs/gateway-scheduled-restart.log 2>&1`
- 最终演练：后台 final drill 触发 stop 后，healthcheck 检测 FAIL，随后通过 force-restart 方案恢复，最终 `Runtime: running`、`RPC probe: ok`；主链路可用性恢复通过。
- 剩余边界：飞书“主动告警送达”在用户前台仍未稳定可见，因此这次验收以“自动恢复后用户能重新正常发消息并收到回复”为准，不再把“通知前台显著收到”作为主链路验收条件。
- 运维规则：
  - 日常人工检查：`openclaw gateway status`
  - 人工重启：`openclaw gateway restart`
  - 极端情况下再用前台调试：`openclaw gateway`
- 注意：本次排查过程中，用户把飞书 `appSecret` 与 gateway token 发到聊天里，后续建议重置这些密钥。
