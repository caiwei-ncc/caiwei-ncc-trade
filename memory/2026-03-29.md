## memory_search 故障最终收口（00:52-01:05）

- 用户提供了当前实际使用的 videnx 中转 key：`sk-3fa4d19e7f407bddd4a50b20255252c76b8555ec4bbd640c89480ae614623899`
- 本轮先核查 `/root/.openclaw/openclaw.json`、环境变量与 systemd gateway 配置，确认：
  - `models.providers.videnx.apiKey` 已是该新 key
  - 当前进程里的 `OPENAI_API_KEY` 也是该新 key
  - 401 的真正原因不是 key 文本写错，而是 `memory_search` 默认走 OpenAI 官方 embeddings 端点，而该 key 属于 videnx 中转，不被 OpenAI 官方端点接受
- 随后按 OpenClaw memory 文档尝试将 `agents.defaults.memorySearch` 显式配置为 `provider=openai`，并把 `remote.baseUrl` 指向 videnx 中转端点，期间为使配置生效，**我手动执行了两次 gateway 重启**；因此用户前台收到了两次“OpenClaw 网关已自动恢复”的恢复通知
- 过程中已验证到 3 个关键事实：
  1. 指向 `https://token.videnx.com/v1/` 时，`memory_search` 从 401 变为 404，说明请求已不再打到 OpenAI 官方端点
  2. 改成 `https://token.videnx.com/` 后，接口返回的是 videnx 网页 HTML，而不是 embeddings JSON
  3. 直接 `curl` 验证也确认：`https://token.videnx.com/v1/embeddings` 返回 `404 page not found`，`https://token.videnx.com/embeddings` 返回网站首页 HTML，不是可用的 embeddings API
- 最终结论：**videnx 当前这条中转只支持 chat / responses，不支持 OpenAI embeddings**；因此 `memory_search` 无法通过该中转修复
